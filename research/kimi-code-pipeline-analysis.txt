### PIPELINE WITH FUNCTION NAMES & CALL FLOW

**Stage 1: Initialization (SessionStart hook)**
- Binary loads plugin registry: `/data/data/com.termux/files/home/.kimi-code/plugins/installed.json`
- Reads plugin manifest: `.kimi-plugin/plugin.json` (or older `kimi.plugin.json` for MCP plugins)
- Extracts fields:
  - `skills: "./skills/"` → directory containing SKILL.md files
  - `skillInstructions: <markdown>` → tool-mapping instructions for LLM
  - `sessionStart.skill: "using-superpowers"` → bootstrap skill name

**Stage 2: System Prompt Construction**
- Base prompt: 26698 chars (observed)
- ++ Kimi tool reference (Read, Write, Bash, Grep, Glob, FetchURL, WebSearch, etc.)
- ++ All plugin `skillInstructions` (markdown prose defining tool mappings)
- ++ All skill descriptions from SKILL.md:
  ```
  ---
  name: brainstorming
  description: "Use this before creative work..."
  ---
  ```
- Result: ~50–60KB system context

**Stage 3: Prompt Analysis & Skill Classification (LLM-side)**
1. User submits prompt via UserPromptSubmit hook
2. Binary constructs LLM request: system prompt (50–60KB) + conversation history + user message
3. LLM reads skill descriptions and applies Superpowers' **"using-superpowers" enforcement rule:**
   ```
   If you think there is even a 1% chance a skill might apply, YOU MUST invoke it.
   IF A SKILL APPLIES, YOU DO NOT HAVE A CHOICE.
   ```
   Red-flags table prevents rationalization:
   - Thought: "This is simple" → Reality: "Questions are tasks. Check skills."
   - Thought: "I need context first" → Reality: "Skill check BEFORE context gathering."

**No keyword matching or embeddings detected.** Classification is fully LLM-driven via prompt text.

**Stage 4: Tool Selection (tool-select experimental feature)**
- Config flag: `[experimental] tool-select = true`
- Observable behavior:
  - `toolCount=26` reported to model
  - Multi-step execution: `turSteps 0.1, 0.2, ... 0.8` (8 substeps per turn)
  - Tools available: Bash, Read, Write, Edit, Grep, Glob, FetchURL, WebSearch, AskUserQuestion, TodoList, Agent, Skill, + MCP tools (26+)

**Algorithm (inferred; binary internals opaque):**
- LLM receives all tools in context
- Per substep, LLM selects subset of tools
- Binary orchestrates: `call_llm → parse_tools → execute_tools → gather_outputs → feed_back_to_llm → repeat`
- Possible ranking strategy: logits-based scoring, schema constraints, or learned weights (not visible)

**Stage 5: Tool Execution & Feedback**
- PreToolUse hook fires before each tool call
- Tool executes (e.g., Bash, WebSearch)
- Outcome captured: `{ tool_name, outcome: "success" | "failure", duration_ms }`
- Tool output fed back into next turn's context
- If outcome="failure", retry up to 3 times (max_retries_per_step=3)

**Stage 6: State & Compaction**
- PreCompact hook: if context approaching 50KB budget, compress message history
- Session state tracked: `/session-dir/state.json`
- Wire protocol: `/session-dir/agents/main/wire.jsonl` (all turns with timing)
